# 高性能即时通信服务端 MVP

## 目标

将当前 TalkHub 从“轻量级 WebSocket 聊天系统”调整为“面向简历展示的高性能即时通信系统”。

第一版 MVP 不追求一次性覆盖语音、复杂群组能力和完整分布式一致性，而是优先跑通以下主链路：

- JWT 登录
- Netty 长连接建连
- 单聊 / 单频道消息收发
- 应用层心跳保活
- ACK 确认与重投
- Redis 会话状态与离线消息缓存
- PostgreSQL 消息落库

## MVP 范围

### 必做能力

- 基于 Spring Boot 提供登录、历史消息、管理类 REST API
- 新增独立 Netty 即时通信服务，负责 TCP 长连接、编解码、心跳、ACK、会话管理
- 自定义消息协议，支持消息头、消息体、消息类型、序列号、消息 ID
- 支持 gzip 压缩开关与阈值控制
- 基于 `IdleStateHandler` 实现读空闲 / 写空闲检测、PING/PONG 双向心跳、死链剔除
- 基于消息 ID + ACK 表实现可靠投递与超时重发
- Redis 维护用户在线会话、连接路由、离线消息缓存、待 ACK 状态
- PostgreSQL 持久化用户、会话、消息、ACK 投递日志
- 前端先保留 React + Vite，改为通过专用 TCP 调试客户端、网关适配层或 HTTP 调试接口验证链路

### 暂不纳入第一版

- 语音房间与媒体流
- RabbitMQ 异步广播主链路
- 多机房部署
- 端到端加密
- 完整消息已读回执体系
- 多端漫游同步冲突处理

## 建议的第一版架构

### 服务划分

- `talkhub-api`
  - Spring Boot
  - 负责登录鉴权、历史消息查询、用户与频道元数据管理
- `talkhub-im`
  - Netty
  - 负责长连接、协议编解码、心跳、ACK、会话路由、离线投递
- `Redis`
  - 在线状态
  - 用户会话索引
  - 离线消息缓存
  - 待 ACK 消息索引
- `PostgreSQL`
  - 用户、频道、消息、投递日志

### 连接模型

- 客户端登录后拿 JWT
- 客户端连接 Netty 服务端并携带 JWT
- Netty 鉴权通过后建立 `userId -> channel/session` 绑定
- 服务端返回连接确认帧
- 双方进入心跳保活状态

### 消息主链路

1. 客户端发送业务消息
2. Netty 服务端解码、鉴权、生成消息 ID
3. 消息先写 PostgreSQL
4. 在线用户直接投递
5. 离线用户写入 Redis 离线队列
6. 接收端返回 ACK
7. 服务端清理待 ACK 状态
8. 超时未 ACK 则触发有限次重投

## 协议设计建议

### 帧结构

- `magic`：协议魔数
- `version`：协议版本
- `serializer`：序列化方式
- `compress`：压缩方式
- `messageType`：消息类型
- `requestId`：请求序号
- `messageId`：消息唯一 ID
- `timestamp`：发送时间
- `bodyLength`：消息体长度
- `body`：消息体

### 消息类型

- `AUTH_REQ`
- `AUTH_RESP`
- `CHAT_REQ`
- `CHAT_RESP`
- `ACK_REQ`
- `PING`
- `PONG`
- `ERROR`

### 编解码 MVP 策略

- 默认 JSON 序列化，便于先联调
- 大于阈值的消息体启用 gzip 压缩
- 编解码基于 `ByteBuf`，避免不必要字节数组拷贝
- 第二阶段再切换为 protobuf 或自定义二进制体，进一步压缩带宽与 CPU

## 核心模块

### 1. 连接状态管理

- `ChannelGroup` 或连接注册表维护在线连接
- 用户级连接索引
- 设备级连接索引
- 空闲超时自动下线
- 异常断开自动清理 Redis 会话状态

### 2. ACK 可靠投递

- 服务端为核心消息生成全局 `messageId`
- 投递后写入待 ACK 表
- 在 Redis 中记录 `messageId -> retryMeta`
- 定时扫描超时未 ACK 消息
- 达到最大重试次数后标记投递失败

### 3. 离线消息

- 用户不在线时，消息写入 Redis List / Stream
- 用户重连后拉取并补发
- 补发成功后根据 ACK 删除缓存或标记完成

### 4. 压缩与性能优化

- 编码前按阈值判断是否压缩
- 压缩指标记录：原始字节数、压缩后字节数、压缩耗时
- 解码指标记录：解压耗时、反序列化耗时
- ByteBuf 池化与零拷贝切片优先

## 数据模型增量

在原有 `User / Channel / Message / ChannelMember / OnlinePresence` 基础上，新增：

- `ImSession`
  - 用户会话、设备标识、连接状态、最近心跳时间
- `MessageDelivery`
  - 消息投递状态、ACK 状态、重试次数、最后投递时间
- `OfflineMessageCursor`
  - 用户离线补发进度

## 与最初设计的变化

### 1. 实时通信内核变了

原设计：

- Spring Boot + Netty 长连接服务负责实时通信

新方案：

- Spring Boot 保留为业务 API 层
- 实时链路改为独立 Netty 长连接服务

变化原因：

- 更容易体现非阻塞 I/O、多路复用、连接管理、编解码优化等简历亮点

### 2. “聊天系统”变成“IM 基础设施”

原设计重点：

- 频道聊天
- 历史消息
- 在线状态

新设计重点：

- 连接保活
- ACK 可靠投递
- 离线消息
- 会话路由
- 压缩和编解码性能

变化原因：

- 更接近高性能即时通信服务端的工程叙事，而不是普通业务聊天应用

### 3. RabbitMQ 优先级下降

原设计：

- RabbitMQ 作为异步落库和广播关键组件

MVP 新设计：

- 第一版先用 Netty + Redis + PostgreSQL 跑通可靠消息链路
- RabbitMQ 放到第二阶段，用于削峰、跨节点广播和异步持久化优化

变化原因：

- 先把最能体现 IM 核心能力的链路做实，减少首版系统复杂度

### 4. Redis 角色变重

原设计：

- 缓存在线状态、成员态、未读数

新设计：

- 除在线状态外，还承担会话路由、待 ACK 状态、离线消息缓存

变化原因：

- 这是支撑“分布式会话管理”和“精准路由”的关键组件

### 5. 协议层从文本消息升级为自定义帧协议

原设计：

- 更偏业务层文本消息收发

新设计：

- 引入协议头、消息类型、压缩位、序列号、ACK 语义

变化原因：

- 简历中的压缩比、编解码、可靠投递都依赖协议层控制能力

## 第一版交付标准

满足以下条件即可作为第一版 MVP：

- 单机可稳定维护大量长连接
- 用户可通过 JWT 建立 IM 长连接
- 服务端支持心跳保活与死链自动清理
- 单聊消息支持 ACK、超时重投、离线补发
- 消息可落 PostgreSQL，并可查询历史记录
- Redis 中可观测在线会话、离线消息和待 ACK 数据
- 压缩功能可开关，并能输出基础性能统计

## 第二阶段再补的简历增强点

- protobuf / 自定义二进制序列化替换 JSON
- 本地压测脚本与指标看板
- RabbitMQ 跨节点广播与削峰
- 多节点部署下的用户路由一致性
- 批量 ACK、滑动窗口重传、消息去重
- 真正可量化的压缩比、编解码耗时、CPU / GC 指标
