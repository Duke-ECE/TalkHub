# TalkHub AGENTS Guide

本文件是 TalkHub 项目的全局协作与长期记忆约束。后续在本仓库内进行分析、设计、编码、测试、文档更新时，默认遵守本文件，除非用户在当前任务中明确覆盖。

当前用户已明确要求放弃 `WebSocket` 方案，改为 `Netty` 长连接即时通信架构。后续实现、文档与设计均以该决定为准。

## 1. 文档优先级

项目当前有两份关键来源文档：

- `specs/配置规定.md`
- `specs/项目总需求.md`

发生冲突时，按以下顺序裁定：

1. 用户在当前对话中的明确要求
2. `specs/配置规定.md`
3. `specs/项目总需求.md`
4. 代码仓库中的既有实现

## 2. 项目定位

TalkHub 是一个轻量级实时聊天与语音平台，目标能力包括：

- 实时频道聊天
- 成员状态同步
- 在线用户管理
- 历史消息查询
- JWT 鉴权
- 统一异常处理
- 接口规范化

## 3. 当前确认的技术栈

### 前端

- 使用 `React + Vite`
- 使用 `Tailwind CSS` 组件体系

### 后端

- 以 `Spring Boot` 作为业务 API 层
- 使用 `Netty` 承担实时通信与长连接管理
- 使用 `Redis` 缓存在线状态、会话路由、离线消息与待 ACK 状态
- 使用 `RabbitMQ` 处理后续阶段的消息异步落库与跨节点事件广播

### 数据库

- 统一按 `PostgreSQL` 实现
- 数据库连接通过环境变量配置

以下环境变量是当前已确认的数据库配置来源：

```env
DB_DSN=
PG_HOST=localhost
PG_PORT=5432
PG_USER=postgres
PG_PASSWORD=postgres
PG_DATABASE=echocenter
PG_SSLMODE=disable
```

## 4. 冲突裁定记录

`docs/项目总需求.md` 中提到 `MySQL`，但 `docs/配置规定.md` 明确规定数据库使用 `PostgreSQL`。后续所有设计、代码、脚本、文档、SQL 方言、ORM 配置、容器配置，一律以 `PostgreSQL` 为准，不再按 `MySQL` 落地。

如需恢复为 `MySQL`，必须由用户明确提出，并同步更新配置文档。

## 5. 默认架构理解

若用户未另行指定，按以下方向组织系统：

- 前端：React 单页应用
- 后端：Spring Boot 提供 REST API，Netty 提供 IM 长连接服务
- Redis：缓存在线态、会话态、离线消息、待 ACK 数据
- RabbitMQ：第二阶段用于削峰与异步事件分发
- PostgreSQL：持久化用户、频道、消息、关系数据

## 6. 领域模型最小集合

后续设计与实现时，默认至少覆盖这些核心实体：

- User
- Channel
- ChannelMember
- Message
- MessageReadState 或 UnreadCounter
- OnlinePresence
- ImSession
- MessageDelivery

如果新增实体，应优先说明其与上述核心模型的关系。

## 7. 开发约束

- 前端新增页面或组件时，优先采用 React 函数组件与 Vite 工程结构
- 前端样式优先使用 Tailwind 工具类或基于 Tailwind 的组件封装
- 后端接口应保持清晰的 DTO、统一异常响应和鉴权边界
- 实时能力优先考虑连接状态、订阅粒度、消息顺序、幂等与断线重连
- 涉及缓存时，必须明确缓存键、过期策略、回源逻辑与一致性风险
- 涉及 RabbitMQ 时，必须明确生产者、消费者、消息体结构、重试和死信策略
- 涉及 PostgreSQL 时，避免写入 MySQL 专属语法
- 所有新增环境变量都应记录到文档或示例环境文件中

## 8. 实现优先级建议

若用户没有指定顺序，默认按以下阶段推进：

1. 初始化工程结构与基础配置
2. 用户鉴权与基础用户模型
3. 频道与成员管理
4. Netty 即时通信主链路
5. 消息持久化与历史查询
6. 在线状态、未读数、缓存策略
7. RabbitMQ 异步化与广播优化
8. 测试、部署与文档完善

## 9. 文档与代码同步规则

- 发现需求与配置冲突时，先在实现中遵循高优先级文档，再把冲突写明
- 引入新的基础设施、中间件、端口、环境变量、脚本时，应同步更新相关文档
- 如果实现必须偏离本文件，需在交付说明中明确说明偏离点和原因

## 10. 对后续代理的要求

后续任何代理在本仓库工作时，应先把本文件视为项目常驻记忆，再结合当前任务追加局部决策。不要因为旧经验默认使用其他前端栈、数据库或消息中间件。
